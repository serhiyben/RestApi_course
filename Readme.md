## Що реалізовано в Лабі №7: Rate Limiter (Обмеження запитів)
У цій лабораторній роботі реалізовано механізм Rate Limiting для захисту API від надмірного навантаження (спаму, парсерів, DDoS) з використанням in-memory сховища Redis.

### Основні зміни та досягнення:
1. **Інтеграція Redis та Sliding Time Window:** Додано асинхронний клієнт `redis` для зберігання історії запитів. Реалізовано точний алгоритм "Sliding time window" (ковзне вікно часу), який відстежує запити з точністю до часток секунди, динамічно очищаючи застарілі дані за допомогою методів `zremrangebyscore` та `zadd`. У `docker-compose.yml` додано окремий сервіс для Redis.
2. **Диференційовані ліміти (Role-based Limits):** Налаштовано гнучкі рівні доступу. Для анонімних користувачів (ідентифікація за IP-адресою) встановлено суворий ліміт: **2 запити на хвилину**. Для авторизованих користувачів (ідентифікація за JWT-токеном) ліміт розширено до **10 запитів на хвилину**. При перевищенні квоти API коректно повертає статус `429 Too Many Requests`.
3. **Адаптація маршрутів (Endpoints):** Ендпоінти оновлено для прийому об'єкта `Request`. Створено окремий публічний маршрут `GET /books/public` з підтримкою пагінації для демонстрації роботи анонімних лімітів. Основні CRUD-операції з книгами залишаються захищеними JWT-авторизацією та підпадають під розширені ліміти.
4. **Unit-тестування з моканням (Mocking):** Написано ізольовані тести для перевірки логіки Rate Limiter. Щоб не навантажувати реальний сервер під час тестів, використано `unittest.mock.patch` та `AsyncMock` для імітації відповідей від Redis. Покрито всі 4 обов'язкові тест-кейси: успішний запит (200) та блокування (429) окремо для анонімного та авторизованого користувача, зі збереженням стабільного Event Loop.

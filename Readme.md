## Що реалізовано в Лабі №8: Rate Limiter (Обмеження запитів)
У цій роботі ми створили Mock-сервер, який імітує роботу нашої бібліотеки. Це дозволяє розробникам фронтенду або мобільних додатків працювати з API ще до того, як бекенд чи база даних будуть повністю готові.


### Основні зміни та досягнення (Лабораторна робота №8):

1.  **Розробка OpenAPI специфікації (OAS):** Створено повний технічний опис API бібліотеки у файлі `openapi.yaml`. Специфікація стандартизує структуру проекту, описуючи всі доступні маршрути, методи (GET, POST, DELETE), необхідні параметри запитів та структуру JSON-відповідей. Це дозволяє використовувати файл як «єдине джерело істини» для всієї команди розробників.
2.  **Контейнеризація Mock-сервера через Stoplight Prism:** У `docker-compose.yml` додано сервіс на базі інструменту Prism. Налаштовано автоматичне розгортання ізольованого Mock-сервера на порту `4010`. Реалізовано монтування волюмів (`volumes`) для синхронізації файлу специфікації між локальною машиною та контейнером, що забезпечує актуальність Mock-даних без перезбірки образу.
3.  **Симуляція відповідей (Data Mocking):** Для кожного ендпоінту (реєстрація, отримання токенів, робота з книгами) розроблено блоки `example`. Це дозволяє Mock-серверу повертати передбачувані статичні дані, що імітують успішне виконання операцій. Такий підхід забезпечує незалежність розробки фронтенд-частини від стану реального бекенду та бази даних.
4.  **Емуляція механізмів безпеки (Security Schemes):** У специфікації реалізовано схему `bearerAuth` для JWT-токенів. Mock-сервер налаштований на валідацію вхідних запитів: при спробі звернення до захищених ресурсів (наприклад, перегляд списку книг) без відповідного заголовка `Authorization`, Prism автоматично повертає помилку `401 Unauthorized`, повністю імітуючи логіку безпеки реального API.


## Лабораторна робота №8: Mock API з використанням Stoplight Prism

У цій роботі ми створили **Mock-сервер**, який імітує роботу нашої бібліотеки. Це дозволяє розробникам фронтенду або мобільних додатків працювати з API ще до того, як бекенд чи база даних будуть повністю готові.

###  Як запустити Mock-сервер
Всі налаштування вже додані до `docker-compose.yml`. Щоб запустити тільки Mock-сервер, виконайте:

```bash
docker-compose up -d prism
```
Сервер буде доступний за адресою: **`http://localhost:4010`**


###  Як тестувати

Ви можете перевірити роботу Mock-сервера за допомогою `curl` або Postman. 

**1. Публічні маршрути (без авторизації):**
```bash
# Отримати список книг для анонімів
curl http://localhost:4010/books/public

# Реєстрація (POST запит)
curl -X POST http://localhost:4010/register
```

**2. Захищені маршрути (вимагають токен):**
Оскільки ми описали безпеку в OpenAPI, Prism перевіряє наявність заголовка `Authorization`. Без нього ви отримаєте помилку **401 Unauthorized**.
```bash
# Отримати повний список книг (з фейковим токеном)
curl -H "Authorization: Bearer mock_token" http://localhost:4010/books
```

**3. Видалення ресурсу:**
```bash
# Видалити книгу за ID
curl -I -X DELETE http://localhost:4010/books/123
```

---

###  Важливі зауваження
* **Trailing Slashes:** Mock-сервер чутливий до косої риски в кінці. Використовуйте адреси точно так, як вони описані в `openapi.yaml` (наприклад, `/books/` з слешем в кінці).
* **Статичні дані:** Сервер не зберігає зміни. Якщо ви «видалите» книгу через Mock API, при наступному запиті вона все одно буде в списку, оскільки дані беруться зі статичних прикладів у файлі специфікації.
